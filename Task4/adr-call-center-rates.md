# ADR: Централизованное управление депозитными ставками и их публикация в кол-центры

**Название задачи:** Централизованное управление ставками для MVP депозитов  
**Автор:** Дмитрий  
**Дата:** 2026-03-08

## Контекст

В рамках MVP цифровизации депозитного направления банк запускает новые каналы привлечения клиентов через сайт и интернет-банк.  
При запуске маркетинговой кампании ожидается рост количества звонков в кол-центр. Основная аудитория банка — пожилые клиенты, поэтому значительная часть обращений будет связана с консультациями по депозитным ставкам.

Для обеспечения консультаций:
- внутренний кол-центр банка должен получать доступ к актуальным депозитным ставкам;
- партнёрский кол-центр также должен получать актуальные депозитные ставки;
- партнёрский кол-центр не может использовать API и готов получать данные только в виде файлов.

Текущее состояние неудовлетворительно:
- ставки фактически ведутся в XLS-файлах;
- нет единого централизованного источника правды;
- отсутствует надёжный процесс публикации ставок во все клиентские каналы.

## Решение

Выбирается решение с централизованным сервисом ставок **Deposit Rate Service**.

### Основная идея

Deposit Rate Service становится единым источником правды по депозитным ставкам.  
Сервис хранит ставки, историю изменений и дату публикации, а также предоставляет данные:

- сайту;
- интернет-банку;
- внутреннему кол-центру;
- компоненту файловой выгрузки для партнёрского кол-центра.

### Основные компоненты решения

1. **Rate Management UI / Back-office module**  
   Интерфейс для сотрудников депозитного и кредитного бэк-офиса.

2. **Deposit Rate Service**  
   Прикладной сервис управления ставками, публикации и выдачи ставок потребителям.

3. **Rate DB**  
   База данных ставок, истории изменений, версий публикации и статусов выгрузки.

4. **Rate Export Component**  
   Компонент формирования файловой выгрузки ставок для партнёрского кол-центра.

5. **Call Center Integration**  
   Интеграция внутреннего кол-центра с сервисом ставок.

### Логика работы

- Сотрудники бэк-офиса изменяют ставки через централизованный интерфейс.
- После публикации новая версия ставок становится доступной:
    - сайту,
    - интернет-банку,
    - внутреннему кол-центру.
- Параллельно формируется файл со ставками для партнёрского кол-центра.
- Факт формирования выгрузки и её статус сохраняются в системе.
- На переходном этапе допускается загрузка ставок из XLS в Deposit Rate Service.

### Почему это решение выбрано

- устраняет Excel как рабочий центр процесса;
- синхронизирует все клиентские каналы и кол-центры;
- учитывает ограничение партнёра на отсутствие API;
- минимально влияет на АБС;
- расширяет архитектуру MVP из Task3 без ломки уже выбранной схемы.

## Альтернативы

### Альтернатива 1. Оставить ставки в Excel и вручную рассылать в кол-центры
**Плюсы:**
- быстро и дёшево в моменте.

**Минусы:**
- высокие операционные риски;
- нет единого источника правды;
- высокая вероятность ошибок и устаревших данных;
- не масштабируется.

### Альтернатива 2. Хранить ставки в АБС и оттуда раздавать всем каналам
**Плюсы:**
- единый центр в существующей системе.

**Минусы:**
- АБС уже перегружена;
- растёт зависимость новых каналов от АБС;
- увеличивается риск деградации ядра банка.

### Альтернатива 3. Отдавать ставки партнёрскому кол-центру по API
**Плюсы:**
- быстрее и чище интеграционно.

**Минусы:**
- партнёр не готов к API;
- противоречит ограничению кейса.

## Недостатки и риски

1. Появляется новый прикладной сервис, который нужно сопровождать.
2. На переходном этапе возможна двойственность, пока XLS ещё используется как источник загрузки.
3. Для партнёрского кол-центра остаётся пакетный файловый обмен, а не онлайн-доступ.
4. Нужно отдельно контролировать актуальность файла и успешность его передачи.
5. Потребуется разграничение прав доступа между депозитным и кредитным бэк-офисом.

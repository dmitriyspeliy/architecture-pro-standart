# ADR: Концептуальная архитектура MVP открытия депозитов

**Название задачи:** Концептуальная архитектура MVP открытия депозитов  
**Автор:** Дмитрий  
**Дата:** 2026-03-08

## Функциональные требования

| № | Действующие лица или системы | Use Case | Описание |
| :-: | :- | :- | :- |
| UC-1 | Новый клиент, Сайт | Просмотр депозитных продуктов на сайте | Новый клиент открывает сайт банка и просматривает список доступных депозитов и актуальных ставок |
| UC-2 | Новый клиент, Сайт | Подача заявки на депозит с сайта | Клиент вводит ФИО и номер телефона и отправляет заявку на депозит |
| UC-3 | Сайт, Сервис заявок, Система кол-центра | Передача заявки из сайта в кол-центр | После отправки заявки данные передаются в систему кол-центра для последующего обзвона |
| UC-4 | Оператор кол-центра, Система кол-центра | Обработка заявки нового клиента | Оператор видит заявку, связывается с клиентом и может предложить особые условия |
| UC-5 | Новый клиент, Отделение, АБС | Идентификация нового клиента в отделении | После звонка клиент приходит в отделение для идентификации, так как полное дистанционное открытие для нового клиента недоступно |

| UC-7 | Клиент интернет-банка, Интернет-банк | Подача заявки на депозит из интернет-банка | Клиент выбирает продукт, счёт списания и сумму депозита, затем отправляет заявку |
| UC-8 | Интернет-банк, SMS-сервис | Подтверждение заявки по SMS-коду | Для подачи заявки клиент подтверждает операцию одноразовым SMS-кодом |
| UC-9 | Сервис заявок, АБС, Бэк-офис депозитов | Передача заявки в обработку бэк-офису | На этапе MVP заявка не исполняется автоматически, а передаётся сотруднику бэк-офиса для подтверждения условий |
| UC-10 | Бэк-офис депозитов, АБС | Подтверждение условий депозита | Сотрудник бэк-офиса подтверждает условия депозита в АБС |

| UC-14 | Сервис ставок, АБС | Использование актуальных ставок в процессе обработки заявки | Актуальные ставки и согласованные специальные условия доступны бэк-офису при обработке заявки |
| UC-15 | Интернет-банк, Сервис заявок | Быстрый отклик клиенту при подаче заявки | Клиент получает быстрый ответ о принятии заявки без прямой синхронной нагрузки на АБС |

## Нефункциональные требования

| № | Требование |
| :-: | :- |
| NFR-1 | Все новые внешние взаимодействия должны использовать шифрование трафика (TLS) |
| NFR-2 | Чувствительные данные клиента должны передаваться и храниться в защищённом виде в пределах корпоративных стандартов |
| NFR-3 | Целевой отклик пользовательских операций в цифровых каналах должен составлять миллисекунды |
| NFR-4 | Нельзя строить новый процесс на прямом синхронном API-вызове интернет-банка в АБС |
| NFR-5 | Нельзя увеличивать прямую онлайн-нагрузку на БД АБС, так как она уже перегружена |
| NFR-6 | Новые сервисы должны быть доступны 24/7 с целевой доступностью 99,9% |
| NFR-7 | Решение должно поддерживать горизонтальное масштабирование цифрового контура |
| NFR-8 | Следует максимально использовать технологии, уже существующие в банке: Java, Oracle, MS SQL, существующие платформы |
| NFR-9 | Для асинхронных интеграций желательно использовать Kafka на перспективу, но MVP не должен зависеть от несовместимости текущего монолита интернет-банка |
| NFR-10 | Логи, аудит и технический мониторинг должны быть доступны для новых компонентов |
| NFR-11 | Решение должно быть документировано и пригодно для дальнейшего расширения |
| NFR-12 | Интерфейсы сайта и интернет-банка должны соответствовать корпоративной дизайн-системе |
| NFR-13 | Нужно сохранить возможность последующего поэтапного выделения нового депозитного процесса из монолита интернет-банка |
| NFR-14 | Операции подачи заявки должны быть идемпотентными, чтобы избежать дублей при повторной отправке |
| NFR-15 | Отправка SMS должна быть вынесена в отдельный прикладной сервис, без доработки ядра интернет-банка подрядчиком |

## Решение

Для MVP предлагается реализовать отдельный цифровой контур открытия депозитов, который не нагружает АБС прямыми синхронными вызовами из интернет-банка и сайта.

### Ключевая идея решения

Вместо расширения текущей прямой интеграции интернет-банка с АБС через БД вводится промежуточный слой из новых сервисов:

1. **Deposit Application Service** — прикладной сервис заявок на депозит  
   Отвечает за приём заявок с сайта и интернет-банка, валидацию, регистрацию, хранение статусов, маршрутизацию и интеграцию с внутренними системами.

2. **Deposit Rate Service** — централизованный сервис ставок  
   Хранит актуальные ставки, специальные условия и заменяет Excel как источник ставок для депозитного процесса MVP.

3. **Notification Service** — отдельный сервис уведомлений  
   Реализует отправку SMS и изолирует новый бизнес-функционал от ядра интернет-банка.

4. **Back-office workflow через АБС**  
   На этапе MVP окончательное подтверждение условий и открытие депозита остаётся в АБС и выполняется сотрудником бэк-офиса.

### Общая логика взаимодействия

- **Сайт** и **Интернет-банк** не работают напрямую с АБС по новому сценарию.
- Оба канала вызывают **Deposit Application Service** через защищённый API.
- Интернет-банк перед отправкой заявки инициирует подтверждение через **Notification Service** и SMS-код.
- Заявка сохраняется в отдельном контуре и получает уникальный идентификатор.
- Для получения ставок цифровые каналы и бэк-офис используют **Deposit Rate Service**.
- Deposit Application Service передаёт заявку в АБС через отдельный интеграционный адаптер / шлюз интеграции.
- АБС остаётся системой учёта и системой финального подтверждения на этапе MVP.
- После подтверждения условий и открытия депозита АБС инициирует события/сообщения в Notification Service для отправки SMS.

### Основные архитектурные решения

#### 1. Отделение цифрового контура от АБС
Это нужно, чтобы:
- не увеличивать прямую нагрузку на БД АБС;
- обеспечить быстрый отклик для клиента;
- получить возможность горизонтального масштабирования новых клиентских сценариев.

#### 2. Вынос ставок в отдельный сервис
Это нужно, чтобы:
- убрать Excel и email из основного процесса;
- централизовать управление ставками;
- дать controlled access и депозитному, и кредитному бэк-офису.

#### 3. Вынос SMS в отдельный прикладной сервис
Это нужно, чтобы:
- не дорабатывать ядро интернет-банка подрядчиком;
- переиспользовать единый механизм SMS для сайта и интернет-банка;
- централизовать шаблоны и аудит отправок.

#### 4. Асинхронная интеграция с АБС
Для MVP предпочтителен паттерн:
- заявка принимается в новом контуре;
- затем передаётся в АБС через интеграционный слой;
- клиенту сразу возвращается подтверждение принятия заявки.

Это снижает риск деградации пользовательского интерфейса из-за ограничений АБС.

#### 5. Подготовка к дальнейшему выделению из монолита интернет-банка
Новый функционал депозитов в интернет-банке должен реализовываться как отдельный модуль/адаптер к Deposit Application Service, чтобы в будущем можно было продолжить декомпозицию монолита.

## Альтернативы

### Альтернатива 1. Реализовать MVP полностью внутри интернет-банка и АБС без новых сервисов
**Плюсы:**
- меньше новых компонентов;
- быстрее на бумаге.

**Минусы:**
- усиливается зависимость от подрядческого монолита;
- растёт нагрузка на АБС;
- не решается проблема со ставками в Excel;
- сложно масштабировать и развивать.

### Альтернатива 2. Реализовать всё внутри АБС
**Плюсы:**
- единый контур учёта;
- сотрудники уже работают в АБС.

**Минусы:**
- АБС уже перегружена;
- АБС плохо подходит для быстрого развития цифровых каналов;
- не выполняется требование быстрого отклика;
- возрастает архитектурный риск для основного банковского ядра.

### Альтернатива 3. Сразу строить полноценную event-driven архитектуру на Kafka
**Плюсы:**
- хорошая масштабируемость;
- задел на будущее.

**Минусы:**
- текущая платформа интернет-банка несовместима с Kafka;
- для MVP это увеличивает сроки и сложность;
- есть риск затянуть поставку первой версии.

## Недостатки, ограничения, риски

1. На этапе MVP процесс остаётся частично ручным, потому что подтверждение условий выполняет бэк-офис в АБС.
2. АБС остаётся обязательной системой финального учёта и остаётся узким местом архитектуры.
3. Появляются новые сервисы, а значит увеличивается эксплуатационная сложность.
4. Нужно отдельно продумать консистентность статусов между новым контуром и АБС.
5. Если интеграция с АБС будет реализована неаккуратно, есть риск вернуть старую проблему — фактически синхронную нагрузку на АБС.
6. Отдельно потребуется организационное согласование модели доступа к сервису ставок для депозитного и кредитного бэк-офиса.
7. Для интернет-банка целевые требования 99,9% и полноценная multi-DC устойчивость в MVP могут быть достигнуты только частично, если сам старый монолит останется в одном ЦОД.
8. Notification Service становится важным общим компонентом, поэтому ему нужны мониторинг, ретраи, идемпотентность и контроль доставки.

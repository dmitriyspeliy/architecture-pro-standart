# Task2 — FURPS+ для цифровизации депозитов и кредитов

В таблице ниже выделены архитектурно-значимые требования для MVP и последующего развития цифровых каналов банка «Стандарт».  
Требования собраны по двум направлениям:
- цифровизация депозитов;
- цифровизация кредитов.

Отдельно учтены нефункциональные требования, ограничения текущего IT-ландшафта и дополнительные требования, влияющие на архитектурные решения.

| F | Подача заявки на депозит с сайта для нового клиента: отображение списка депозитов с актуальными ставками, ввод ФИО и телефона, передача заявки в банк | MVP-сценарий для привлечения новых клиентов через сайт |
| F | Подача заявки на депозит из интернет-банка: отображение доступных депозитов и персонализированных ставок, выбор счёта списания и суммы депозита | Ключевой сценарий для действующего клиента |
| F | Подтверждение подачи заявки на депозит по СМС-коду в интернет-банке | Требование безопасности и юридической значимости операции |
| F | Передача заявок с сайта в систему кол-центра для последующего обзвона клиента | Для нового клиента первый контакт происходит через кол-центр |
| F | Возможность менеджера кол-центра просматривать заявку с сайта и предлагать особые условия по депозиту | Это прямо указано в бизнес-процессе |
| F | Возможность сотрудника бэк-офиса депозитов обработать заявку в MVP и подтвердить условия депозита в АБС | На этапе MVP процесс остаётся частично ручным |

| F | Хранение и управление актуальными ставками по депозитам в централизованной системе, доступной депозитному и кредитному бэк-офису | Сейчас ставки ведутся только в XLS, это архитектурно критичная проблема |
| F | Поддержка согласования особых ставок по депозиту без использования Excel и email как основного процесса | Дополнительное архитектурно-значимое требование, потому что ручной процесс мешает масштабированию |
| F | Отображение на сайте списка кредитных продуктов и базовых условий кредитования | Для нового клиента сайт является первой точкой входа |
| F | Подача заявки на кредит с сайта с указанием ФИО и телефона | Базовый MVP-сценарий для нового клиента |
| F | Возможность расширенной заявки на кредит с сайта с вводом паспортных данных для получения предварительного решения | Если данных достаточно, клиент должен сразу получить решение |
| F | Получение предварительного решения по кредиту с сайта на основе данных бюро кредитных историй и скоринга | Для нового клиента доступны только данные БКИ |
| F | Отправка СМС клиенту после подачи кредитной заявки с сайта с приглашением прийти в отделение для идентификации и получения кредита | Это часть процесса для новых клиентов |
| F | Подача заявки на кредит из интернет-банка с выбором счёта зачисления и вводом необходимых данных | Основной цифровой сценарий для действующего клиента |
| F | Отображение в интернет-банке предодобренного предложения по кредиту | Бизнес ожидает работу по модели pre-approved offers |
| F | Повторный скоринг при отправке заявки на кредит из интернет-банка по упрощённой процедуре | Нужно учесть возможное изменение ситуации клиента |
| F | Подтверждение подачи заявки на кредит по СМС-коду в интернет-банке | Требование безопасности и подтверждения операции |
| F | Передача заявок на кредит из сайта и интернет-банка в кредитный конвейер без ожидания суточного batch-обмена через АБС | Архитектурно-значимое требование: иначе нельзя уложиться в обработку в тот же рабочий день |
| F | Возможность сотрудника отделения продолжить работу с уже существующей кредитной заявкой, ранее созданной на сайте | Нужно избежать дублей заявок и потери статуса клиента |
| F | Возможность сотрудника отделения создавать новую кредитную заявку для клиента, если ранее заявка не создавалась | Базовый офлайн-сценарий остаётся |
| F | Отправка клиенту СМС при изменении статуса кредитной заявки | Явное бизнес-требование |
| F | Поддержка единого идентификатора заявки на кредит во всех каналах: сайт, интернет-банк, отделение, кредитный конвейер | Дополнительное требование: без этого сложно реализовать продолжение одной и той же заявки в разных каналах |
| U | Интерфейсы сайта и интернет-банка должны соответствовать корпоративной дизайн-системе банка | Явное требование бизнеса |
| U | Пользователь должен видеть привычные брендированные элементы, цвета и визуальные компоненты | Требование консистентности клиентского опыта |
| U | Клиентский путь подачи заявки на депозит и кредит должен быть максимально коротким, без лишних шагов | Влияет на конверсию и удобство |
| U | Для действующего клиента поля формы должны быть максимально предзаполнены данными интернет-банка | Дополнительное требование, уменьшает время подачи заявки и число ошибок |
| U | Пользователь должен видеть понятные статусы заявки и результаты операций | Иначе растёт нагрузка на кол-центр и снижается доверие |
| U | Ошибки валидации должны отображаться пользователю сразу и в понятной форме | Важно для качества клиентского опыта |
| U | Интерфейс должен быть адаптирован под мобильные устройства | Дополнительное требование, потому что молодая аудитория активно использует мобильный доступ |
| R | Все новые цифровые сервисы должны быть доступны 24/7 | Явное требование бизнеса |
| R | Целевая доступность новых сервисов — не ниже 99,9% | Явное требование бизнеса |
| R | Отказ одного экземпляра приложения или одного узла не должен приводить к остановке всего клиентского процесса | Требование отказоустойчивости |
| R | Сервисы интернет-банка и новых цифровых компонентов должны поддерживать работу в основном и резервном ЦОД | Следует из требований к отказоустойчивости |
| R | При недоступности АБС новые каналы подачи заявок не должны полностью останавливать клиентский фронт | Дополнительное требование: АБС уже перегружена, нужна изоляция отказов |
| R | Должна быть обеспечена гарантированная доставка заявок и уведомлений без потери данных | Критично для банковских операций |
| R | Система должна обеспечивать идемпотентность операций подачи заявки и отправки СМС | Дополнительное требование: предотвращает дубли из-за повторных кликов, таймаутов и ретраев |
| R | Для кредитных заявок должен сохраняться статусный трек и история изменений | Важно для аудита и поддержки клиентского пути |
| P | Отклик пользовательских операций в каналах сайта и интернет-банка должен измеряться миллисекундами | Явное требование бизнеса |
| P | Загрузка справочных данных должна быть существенно быстрее текущего состояния и не занимать более 1 секунды | Сейчас есть проблема со справочниками дольше секунды |
| P | Новые цифровые сервисы должны поддерживать горизонтальное масштабирование | Явное требование, особенно для front-сервисов и кредитных систем |
| P | Запросы из интернет-банка и сайта должны равномерно распределяться между серверами и ЦОД | Явное требование бизнеса |
| P | Нельзя допускать прямой высокой онлайн-нагрузки на АБС со стороны интернет-банка и сайта | Архитектурно-значимое ограничение из-за перегруженной БД АБС |
| P | Обработка заявок по предодобренным кредитам должна позволять принять решение в пределах того же рабочего дня | Иначе бизнес-цель не будет достигнута |
| P | Дополнительные процессы не должны создавать существенную нагрузку на систему кредитного скоринга в рабочие часы | Скоринг уже находится под высокой нагрузкой |
| P | Операции подтверждения по СМС не должны существенно замедлять основной пользовательский сценарий | Дополнительное требование для хорошего UX |
| S | Новые решения должны максимально использовать уже существующие технологии банка: Oracle, MS SQL, Java, существующие платформы | Явное ограничение и требование поддерживаемости |
| S | Предпочтение должно отдаваться технологиям, по которым у банка уже есть внутренняя экспертиза | Это снижает риски внедрения и сопровождения |
| S | Для новых интеграций должна быть подготовлена документация, достаточная для дальнейшего расширения системы | Явное требование из задания |
| S | Новые решения должны быть совместимы с существующими платформами разработки банка и подрядчиков | Требование для реальной интеграции в текущий ландшафт |
| S | При необходимости использования очередей сообщений предпочтительно использовать Kafka | Явное технологическое предпочтение на перспективу |
| S | Архитектура должна позволять поэтапно выделять новый функционал интернет-банка по депозитам и кредитам в отдельные сервисы | Дополнительное требование: это снижает зависимость от монолита и подрядчика |
| S | Решение должно предусматривать централизованное логирование, мониторинг и трассировку новых компонентов | Дополнительное требование для эксплуатации 24/7 и быстрого поиска инцидентов |
| S | API и события новых сервисов должны быть версионируемыми | Дополнительное требование: пригодится при дальнейшем развитии каналов и интеграций |
| +R | Чувствительные данные, передаваемые с сайта и интернет-банка, должны быть защищены шифрованием трафика | Явное требование, фактически TLS для внешних и внутренних защищённых соединений |
| +R | Нельзя дорабатывать СМС-функциональность в ядре интернет-банка силами подрядчика, если это можно реализовать силами команды банка | Явное организационно-технологическое ограничение |
| +R | Текущая версия платформы интернет-банка несовместима с Kafka | Ограничение, влияющее на выбор способа интеграции для MVP |
| +R | АБС не должна становиться синхронной точкой высокой нагрузки в новых цифровых процессах | Ограничение из-за перегруженной БД АБС |
| +R | АБС масштабируется только вертикально | Жёсткое инфраструктурное ограничение |
| +R | Интернет-банк сейчас фактически работает в одном ЦОД и не соответствует целевым требованиям доступности | Ограничение текущего состояния, которое нужно учитывать при проектировании MVP |
| +R | Система скоринга не должна использоваться для массовых предрасчётов предодобренных предложений в рабочие часы | Явное ограничение из задания |
| +R | Для новых клиентов при работе с кредитной заявкой до идентификации можно использовать только доступные внешние и общие данные, прежде всего БКИ | Ограничение бизнес-процесса и данных |
| +R | Новый клиент не может получить продукт полностью онлайн без посещения отделения для идентификации | Жёсткое регуляторное ограничение |
| +R | На этапе MVP заявка на депозит подтверждается менеджером бэк-офиса в АБС, а не полностью автоматически | Ограничение рамок MVP |
| +R | На этапе MVP необходимо избежать глубокого рефакторинга всего интернет-банка, но заложить основу для дальнейшего выделения микросервисов | Дополнительное требование, вытекает из ограничений по срокам и платформе |

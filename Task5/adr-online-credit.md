# ADR: Архитектура онлайн-подачи заявки на кредит

**Название задачи:** Онлайн-заявка на кредит  
**Автор:** Дмитрий  
**Дата:** 2026-03-08

## Контекст

Банк «Стандарт» планирует в течение года после MVP депозитов реализовать цифровой процесс подачи кредитной заявки через сайт и интернет-банк.

Текущее состояние процесса не подходит для целевого сценария:
- кредитная заявка подаётся только в отделении;
- заявка из АБС попадает в Кредитный конвейер пакетно раз в сутки;
- финальный статус также возвращается в АБС пакетно;
- такая схема не позволяет обрабатывать онлайн-заявки в течение того же рабочего дня;
- система скоринга уже находится под высокой нагрузкой и не должна быть перегружена дополнительными фоновыми расчётами.

При этом:
- новый клиент на сайте может указать ФИО, телефон и, опционально, паспортные данные;
- если данных достаточно, клиент должен получить предварительное решение;
- действующий клиент в интернет-банке может видеть кредитные продукты и предодобренные предложения;
- после подачи заявки должен выполняться повторный скоринг по упрощённой процедуре;
- заявка должна подтверждаться SMS-кодом;
- если клиент потом приходит в отделение, сотрудник должен продолжить работу с той же заявкой, а не создавать дубликат.

## Решение

### Ключевая идея

Для онлайн-кредитования вводится отдельный цифровой контур **Credit Application Service**, который принимает заявки из сайта и интернет-банка, инициирует скоринг, сохраняет статусы, отправляет уведомления и интегрируется с Кредитным конвейером напрямую, минуя суточный batch-путь через АБС.

АБС остаётся:
- системой работы сотрудника отделения;
- системой выдачи кредита и учёта операций;
- точкой продолжения процесса в отделении.

Но она перестаёт быть единственной входной точкой кредитной заявки для цифровых каналов.

### Основные компоненты решения

1. **Credit Application Service**  
   Новый прикладной сервис приёма, валидации, регистрации и маршрутизации кредитных заявок.

2. **Credit Application DB**  
   Хранение заявок, статусов, идентификаторов, истории изменений, idempotency-records.

3. **Offer/Pre-Approval Component**  
   Компонент отображения доступных предодобренных предложений для интернет-банка.

4. **Scoring Orchestrator**  
   Компонент orchestration для вызова скоринга по нужному сценарию:
    - новый клиент,
    - действующий клиент,
    - повторный скоринг по упрощённой процедуре.

5. **Notification Service**  
   Отправка SMS-кодов и SMS по статусам заявки.

6. **Credit Conveyor Integration Adapter**  
   Адаптер интеграции с Кредитным конвейером в near real-time.

7. **ABS Integration Adapter**  
   Адаптер для доступа отделений к существующей цифровой заявке и синхронизации статусов с АБС.

### Основные принципы решения

#### 1. Отделение цифрового канала от АБС
Сайт и интернет-банк не должны ждать суточного batch и не должны создавать нагрузку на АБС как на точку входа.

#### 2. Direct integration с Credit Conveyor
Новый цифровой контур должен передавать заявку напрямую в Кредитный конвейер через отдельный адаптер, а не через старый процесс АБС → batch → конвейер.

#### 3. Централизованное управление заявкой
Заявка создаётся один раз и получает единый идентификатор.  
Этот идентификатор используется:
- в сайте,
- в интернет-банке,
- в цифровом контуре,
- в кредитном конвейере,
- в отделении.

#### 4. Контролируемое использование скоринга
Скоринг вызывается по запросу для конкретной заявки.  
Массовые предрасчёты в рабочее время не должны перегружать scoring-систему.

#### 5. Разделение онлайн-решения и финального оформления
Предварительное решение по онлайн-заявке и окончательное оформление/выдача кредита — разные этапы.  
Для нового клиента посещение отделения для идентификации остаётся обязательным.

### Как работает решение

#### Сценарий 1. Новый клиент с сайта
- Клиент заполняет заявку.
- Credit Application Service создаёт заявку.
- Если данных достаточно, Scoring Orchestrator инициирует скоринг.
- Используются данные БКИ и доступные общие данные.
- Клиент получает предварительное решение или статус принятия заявки.
- Заявка уходит в Credit Conveyor.
- Клиент получает SMS.
- В отделении сотрудник продолжает работу с той же заявкой.

#### Сценарий 2. Клиент интернет-банка
- Клиент видит продукты и предодобренное предложение.
- Подаёт заявку и подтверждает её по SMS.
- Credit Application Service создаёт/обновляет заявку.
- Выполняется повторный скоринг по упрощённой процедуре.
- Заявка уходит в Credit Conveyor.
- Статусы и уведомления обновляются через цифровой контур.

### Почему выбрано именно это решение

- выполняет требование обработки заявок быстрее, чем batch раз в сутки;
- не перегружает АБС как онлайн-точку входа;
- позволяет сохранить Кредитный конвейер как основную систему обработки заявок;
- сохраняет возможность дальнейшего роста цифрового контура;
- поддерживает единый клиентский путь между онлайн-каналом и отделением.

## Альтернативы

### Альтернатива 1. Оставить вход только через АБС
**Плюсы:**
- меньше новых компонентов;
- проще встроиться в текущую модель.

**Минусы:**
- сохраняется batch раз в сутки;
- не выполняется бизнес-требование по скорости;
- АБС остаётся узким местом.

### Альтернатива 2. Встроить весь новый процесс в Кредитный конвейер
**Плюсы:**
- конвейер уже является целевой системой обработки кредитных заявок;
- меньше промежуточных компонентов.

**Минусы:**
- в нём нет полноценного цифрового фасада для сайта и интернет-банка;
- сложнее решать задачи SMS-подтверждения, идемпотентности, UI-ориентированных статусов;
- сильнее связывает клиентские каналы с бэк-офисной системой.

### Альтернатива 3. Сразу строить полностью event-driven архитектуру на Kafka
**Плюсы:**
- хороший задел на будущее;
- удобная масштабируемость.

**Минусы:**
- увеличивает сложность внедрения;
- текущая платформа интернет-банка ограничивает прямую интеграцию;
- для задачи одного года может быть избыточно.

## Ограничения и недостатки решения

1. Появляется новый цифровой слой, который нужно сопровождать отдельно.
2. Требуется аккуратная синхронизация статусов между Credit Application Service, Credit Conveyor и АБС.
3. АБС не исключается полностью из процесса, так как отделение и выдача кредита остаются в ней.
4. Система скоринга остаётся критичным и чувствительным к нагрузке компонентом.
5. Предодобренные предложения требуют отдельного механизма расчёта/обновления, который здесь не детализирован глубоко.
6. Для нового клиента окончательное оформление кредита всё равно требует визита в отделение.
7. При плохой реализации адаптеров есть риск получить дубли заявок или расхождение статусов.
